# Skill-router subprocess and file-read effects cross injected seams

Status: accepted

`skill-router` made every external effect a bare leaf call: three `Bun.spawn` sites (`bunx @tanstack/intent list`/`load`, `git rev-parse`) plus direct `Bun.file` reads of config and skill bodies. The only way to observe them was from *outside* the interface — the gating test forged an on-disk `#!/bin/sh` runner, `chmod`'d it, grepped a log file to prove "did Intent spawn?", and mutated `process.env.HOME` so config discovery found the fixture. We deepened those effects behind a **Router runtime** (`RouterRuntime`) that carries the **Command seam** (`run(argv) → { exitCode, stdout }`, production `bunCommandRunner`), the **Skill-read seam** (`read(path) → string | null`, production `bunReadText`), the process `env`, and an optional `configPath`. Tests inject a recording runner plus a controlled runtime, so "did the pinned runner spawn, with what argv?", "which HOME expanded?", and "did package load read the returned body?" become assertable *at the interface*, and the shell-script/log-file fixture plus HOME swapping are deleted. The seams express only *how* effects happen, never *when*: ADR-0006's `includePackage` gate and `@0.0.41` pin stay above them, and the production spawn adapter **refuses to run under `bun test`** (a `bunfig.toml` preload sets a guard env var) so a forgotten injection fails loud and offline instead of making a real `bunx @tanstack/intent` network call.

## Considered options

- **Two domain ports (`IntentPort.list/load` + `GitPort.findRoot`) instead of one generic spawn port.** A deeper interface per method, but hiding argv inside `list()` means the test double can no longer see the pinned `@0.0.41` binary — exactly the ADR-0006 fact worth asserting — and it forces the double to re-implement Intent's JSON contract. Rejected: more depth, worse at the assertion that matters. (A four-design panel scored the generic port highest on locality + ADR-fidelity precisely here.)
- **Bind every effect into one Router runtime, resolved once at the edge as a Router context.** Chosen after the first seam landed: the `RouterRuntime` carries the effect adapters + `env` + `configPath`, and `resolveContext` reads/merges config a single time at the `cli.ts` edge into a `RouterContext = { runtime, config }` threaded through discovery, catalog, load, sync, and install. This keeps `git.ts` dependent only on the `run` capability, removes the test-only `HOME` mutation, and resolving once eliminated the prior double-read (a `catalog`/`install` command parsed config twice). The alternative — threading the runtime but letting each consumer call `loadConfig` — was rejected: it keeps env reachable per-consumer and re-reads config. Not turned into a virtual filesystem.
- **Plain module default with no test guard.** Simpler, but once the on-disk runner fixture is gone a future un-injected test silently makes a real `bunx @tanstack/intent` network call. Rejected: the guard is a one-time cost that keeps the suite hermetic and offline.
- **Fold all file I/O behind the read seam (a virtual filesystem).** Rejected: the Skill-read seam covers config reads and resolved skill-body reads, but directory discovery still walks the real filesystem. That preserves a small interface while removing the globals that made tests leak.

## Consequences

- Every test that reaches the discovery or load path must inject a `RouterContext` (via `resolveContext` over a fixture runtime) — the `git` probe rides the same seam — enforced by the preload guard; there is no silent fallthrough to a real spawn. `loadConfig` is called only by `resolveContext`; downstream modules read `ctx.config`.
- The suite is now fully in-memory (~20ms, down from ~970ms) and hermetic, which unlocks running `bun test` inside `nix flake check` — previously impossible because the tests spawned real processes. Wiring that gate is tracked separately.
- `Bun.spawn` exists at exactly one site (`bunCommandRunner`); the argv-split, exit-code, JSON-parse, and offline→`[]`/`null` rules stay as plain code above the seam. ENOENT now collapses to exit 127 inside the adapter, so callers need no `try/catch` — a testable strengthening of ADR-0006's "never throws offline".
- New vocabulary lives in `CONTEXT.md` ("Router runtime", "Router context", "Command seam", "Skill-read seam", the `intentRunner` / `runnerCmd` disambiguation). Re-introducing a bare `Bun.spawn` outside the seam, calling `loadConfig` from a downstream module, making tests mutate `process.env.HOME` for router config, or removing the under-test guard is a regression.
